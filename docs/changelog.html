Changelog & Roadmap

Release history and planned work for the Vernon Project app. Versions run from the initial 0.0.1 baseline to 0.4.0, which adds the Project Admin role. Every release in the source CHANGELOG.md is dated 2026-03-15.

Note

The app's declared package version is 0.0.1, while the changelog tracks feature milestones up to 0.4.0. The dates below are taken verbatim from the source changelog.

Releases

Releases are listed newest first. Each entry lists the changes recorded in CHANGELOG.md.

v0.5.0 — Scoring, allocations & Brand 0.5.0 2026-06-20

  • Project Todo is now a standalone DocType (was a child table of Project Detail). It links its parent via project_detail and owns an allocations child table of Project Todo Allocation rows for per-day time planning.
  • Point-based scoring. Added Group (with weighting + late/early penalties) and its Group Level scale; Project Todo now carries group, level, and point. Awards are recorded in the new Point Ledger, one row per role (assignee / leader).
  • Customer renamed to Brand (field brand_name); Project's customer link is now brand. Brand permissions aligned with the UI gate (Group Manager + Project Owner can manage).
  • Project Detail editing. The mobile create/edit forms now expose keterangan_di_sow, discount, and price (with total = price − discount recomputed server-side), and the previously-required grouping was dropped from those forms (it remains an optional default-group hint).
  • Removed the Project Group DocType and the Project's project_group link.
  • New ERD page and a refreshed data model covering all 14 DocTypes.

See the entity-relationship diagram for the full current model.

v0.4.0 — Project Admin role 0.4.0 2026-03-15

  • Added the Project Admin role and a project_admin field on Project; the admin is auto-added to team members.
  • Project Admin can Read / Write / Create Project Detail and Project Todo, but cannot update todo status.
  • Status-update block enforced at both the API (update_status) and DocType (validate_project_admin_status_update) levels.
  • Project Admin added to get_permission_query_conditions and has_permission for consistent data filtering and document-level access.
  • Security: multi-layer validation to prevent unauthorized status updates.

See Permissions for the full role model.

v0.3.0 — Per-phase time tracking 0.3.0 2026-03-15

  • Added per-phase time-estimation fields: estimated_planned_to_done, estimated_done_to_checked, estimated_checked_to_completed (hours), with an auto-calculated total.
  • Added per-phase actual-time fields: actual_planned_to_done, actual_done_to_checked, actual_checked_to_completed (hours), with an auto-calculated total.
  • Added phase-transition timestamps recorded automatically on each status change.
  • Workflow now records estimated vs. actual time per phase, enabling reporting and analytics for efficiency and bottleneck identification.
  • Added positive-value validation for estimates; the legacy estimated (minutes) field is retained for backward compatibility.

See phase tracking for details.

v0.2.0 — Done-todo field locking 0.2.0 2026-03-15

  • Made assigned_to, estimated, and deadline read-only once a todo is 🟠 Done or ✅ Completed.
  • Added DocType-level validation to reject changes to those protected fields.
  • Security: improved data integrity and protection of historical todo data.

v0.1.0 — Recurring todos 0.1.0 2026-03-15

  • Added the Recurring Todo feature: is_recurring, recurring_frequency (Daily / Weekly / Monthly), recurring_until, next_occurrence, and original_todo fields.
  • Auto-create recurring todos on completion, plus a daily scheduled job (create_recurring_todos) for missed occurrences.
  • Preserves todo details (assigned_to, estimated, notes) across the recurring chain.
  • Extended Project Todo on_change logic and next-occurrence calculation / validation.

v0.0.1 — Initial release 0.0.1 2026-03-15

  • Initial release of the Frappe-based project-management system.
  • DocTypes: Project, Project Detail, Project Todo, Customer, Glossary, Project Group, Project Team, Project Glossary, plus basic Scope of Work and Project Proposal.
  • Features: tree-based project grouping, the four-stage todo workflow, role-based team management, progress / estimate tracking, auto status updates, Calendar / Gantt views, business-logic validation, and duplicate team-member removal.
  • API endpoints: get_project_team_members, get_notes, update_status, save_notes.
  • Reports: Progress, Todo, Project Todo Deadline, Daily Assignment, Daily Performance.
  • Permission system: System Manager / Project Owner / Project Leader / Project Team with user-based filtering and workflow validation.
  • Utilities and helper / validation functions.
  • Security: RBAC, team-membership permission filters, API permission checks, and workflow validation.

Documentation vs. implementation

The repository's README.md documents the 0.0.1 baseline, so a few points in older docs do not yet reflect the current code. This section reconciles those differences so readers know which description to trust.

Known doc / code drift

Where the README and the implemented code disagree, the code is authoritative. The points below are factual differences, not bugs.

  • Features missing from the README. The README does not mention the Project Admin role / project_admin field (v0.4.0), the recurring-todo fields and the create_recurring_todos scheduled job (v0.1.0), the Done / Completed field-locking rule (v0.2.0), or the per-phase estimate and actual-time fields (v0.3.0) — even though these exist in project.json, project_todo.json, and hooks.py. It documents only the 0.0.1 baseline.
  • Audit-field naming mismatch. The README describes Project Todo audit fields named developed_at / developed_by, tested_at / tested_by, and completed_at / completed_by. The phase-tracking implementation instead uses planned_started_at, done_started_at, checked_started_at, and phase_completed_at, with no *_by user fields. The two documents describe different field sets for the same DocType.
  • Report status-filter wording. The README describes Progress Report as showing todos whose status is not Scheduled and Todo Report as Ongoing / Scheduled todos. Those values are not valid Project Todo statuses — the todo workflow uses ⚪️ Planned, 🟠 Done, 🔷 Checked By PL, and ✅ Completed (Pending / Ongoing / Completed are Project Detail statuses). The filter wording does not line up with the todo status model.
  • Incomplete update_status description. The README's update_status walkthrough does not mention the API-level Project Admin status-update block added in v0.4.0, so the documented workflow is incomplete versus the implemented permission checks.
For current behavior

See Project Todo for the implemented fields, Workflow for phase tracking, and Reports for the actual report filters.
