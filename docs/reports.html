Reports & Dashboards

Vernon Project ships five Script Reports and one default Dashboard, all in the Vernon Project module. Three reports render a calendar matrix of estimated hours by deadline day; two are flat audit listings. Every report uses the emoji status values verbatim — ⚪️ Planned, 🟠 Done, 🔷 Checked By PL, ✅ Completed.

Quick reference

All five reports are Script Reports (is_standard='Yes', module Vernon Project, add_total_row=1). Four reference the Project Detail doctype; Todo Report references the standard Frappe ToDo doctype.

Daily Assignment Report

A per-assignee daily work board. It shows all non-pending Project Todos assigned to a chosen user at a given status, laid out as a calendar matrix where each todo's estimated hours appear under its deadline day. It supports inline "ongoing" toggling, status-advance ("Next") actions, and per-todo notes editing.

Filters

FilterTypeNotes
status — StatusSelectOptions: blank, ⚪️ Planned, 🟠 Done, 🔷 Checked By PL, ✅ Completed.
assigned_to — Assigned ToLink · Userrequired Defaults to frappe.session.user.

Columns

Eight fixed columns, followed by one dynamic Int column per calendar day from the earliest to the latest deadline in the result set.

  • ongoing — Check rendered as a checkbox toggle by the report JS.
  • todo — Data; prefixed when the row is marked ongoing.
  • project — HTML anchor link to /app/project/<project_name>.
  • detail_project — HTML anchor link to /app/project-detail/<detail_name>.
  • deadline — Date; the JS colors past deadlines red, today orange, future black.
  • status — Data.
  • action — a Next button calling update_status.
  • note — clickable icon (pencil when notes exist, speech otherwise) opening the notes modal.
Dynamic day columns

One Int column is generated per calendar day between the earliest and latest deadline, stepped daily. Each column's fieldname and label are formatted %d/%m. For each row, only the day column matching the todo's deadline holds its estimated hours; all other day cells are empty.

How it works

Parameterized SQL joins tabProject Todo → tabProject Detail (todo.parent = detail.name) → tabProject (detail.project = project.name). The WHERE clause requires todo.deadline IS NOT NULL, detail.is_pending = 0, todo.status = %(status)s, and todo.assigned_to = %(assigned_to)s; results are ordered by deadline, project name, detail title, then estimated. Rows carry hidden fields (todo_name, assigned_to, project_owner, project_leader, notes, detail_name) consumed by the JS layer, which renders the ongoing checkbox (enabled only for the assigned user, persisted via frappe.client.set_value on Project Todo.ongoing), the Next button, and a notes modal that reads/writes through get_notes and save_notes (editable only by the assignee, project owner, or project leader).

Roles: Project Leader Project Owner Project Team

Empty filters do not stop execution

If the filters or assigned_to are blank, the report displays a prompt via msgprint but does not return early — it proceeds with possibly None parameters.

Daily Performance Report

Measures completed work per assignee over a date range. It plots each Project Todo's estimated hours on the day it reached a chosen completion milestone (Done, Checked By PL, or Completed), highlighting work that landed after its deadline.

Filters

FilterTypeNotes
status — StatusSelectOptions 🟠 Done, 🔷 Checked By PL, ✅ Completed; default 🟠 Done. Selects which milestone date field is plotted.
date_range — Date RangeDateRangerequired Default [today-7, today]; maximum span 120 days.
assigned_to — Assigned ToLink · Userrequired Defaults to frappe.session.user.

Columns

Four fixed columns, followed by one dynamic Int column per calendar day across the selected range (day fieldname %Y-%m-%d, label %d/%m, stepped daily).

  • todo — Data.
  • project — HTML anchor link to /app/project/<project_name>.
  • detail_project — HTML anchor link to /app/project-detail/<detail_name>.
  • deadline — Date; rendered red when the milestone landed overdue.
Dynamic day columns

For each row the report sets the single day column matching the row's milestone date to its estimated value (or 0); all other day cells are empty. The JS renders a date-column value red and bold when that day is past the todo's deadline and non-zero, and shows zero or empty estimates in light grey.

How it works

The chosen status maps through STATUS_DATE_FIELD_MAP to a date expression — 🟠 Done → DATE(todo.developed_at), 🔷 Checked By PL → DATE(todo.tested_at), ✅ Completed → DATE(todo.completed_at) (fallback deadline). The SQL joins tabProject Todo → tabProject Detail → tabProject and requires the chosen <date_field> IS NOT NULL, detail.is_pending = 0, todo.assigned_to = %(assigned_to)s, and <date_field> BETWEEN %(from_date)s AND %(to_date)s. The chosen date field is interpolated by name, but the assignee and date bounds remain bound parameters. Per row it computes is_overdue (deadline and milestone date both present and milestone date later than the deadline).

Validation: the report requires filters, a non-blank assigned_to, and date_range; otherwise it calls msgprint and returns empty. If the range exceeds MAX_DATE_RANGE_DAYS (120) it also msgprints and returns empty.

Roles: System Manager Project Owner Project Leader Project Team

Progress Report

A flat audit log of todo progress across projects. It lists todos with their development, testing, and completion timestamps and actors, optionally filtered by project, user, and an activity date range, to review who did what and when.

Filters

FilterTypeNotes
project — ProjectLink · ProjectFilters on pd.project.
user — UserLink · UserDefaults to frappe.session.user; filters on pt.assigned_to.
date_range — Todo Date RangeDate RangeMatches rows where developed_at, tested_at, or completed_at falls in the range (end date made inclusive via add_days(+1)).

Columns

A fixed column set with no dynamic per-day columns.

ColumnType

Deadline (todo_deadline)Date
Assigned To (assigned_to)Data
Estimated (Hours) (estimated)Int

Tested At / By (tested_at, tested_by)Datetime · Data
Completed At / By (completed_at, completed_by)Datetime · Data
Project (project)Data — composite <project> - <project_detail> - <glossary>
Project Detail ID (project_detail_id)Link · Project Detail

How it works

The SQL joins tabProject Todo pt → tabProject Detail pd (pt.parent = pd.name) → tabProject p (pd.project = p.name) → tabGlossary g (pd.grouping = g.name). The base WHERE enforces pd.todo_count > 0 AND pt.status != 'Scheduled'. Dynamic filters add pd.project, pt.assigned_to, and the activity-date condition (a row matches when any of its three activity timestamps fall in the inclusive range). Results are ordered by deadline, detail name, then estimated. Each row emits the todo fields plus the composite project string and project_detail_id.

Roles: System Manager Project Owner Project Leader Project Team

SQL built by string interpolation

The project, user, and date-range filters are inserted into the query via direct f-string interpolation rather than bound parameters. Treat these filter values as un-parameterized when reasoning about query safety.

Project Todo Deadline Report

A deadline calendar matrix for a single project. It shows all non-pending todos at a given status laid out by deadline day, with each todo's estimated hours appearing under its deadline column, to visualize workload distribution and spot overdue items.

Filters

FilterTypeNotes
project — ProjectLink · Projectrequired Link query filtered to Project.status = 'Ongoing'.
status — StatusSelectOptions blank, ⚪️ Planned, 🟠 Done, 🔷 Checked By PL, ✅ Completed; default ⚪️ Planned.
assigned_to — Assigned ToLink · UserOptional.

Columns

Five fixed columns, followed by one dynamic Int column per calendar day from the earliest to the latest deadline (fieldname and label formatted %d/%m, stepped daily).

  • todo — Data.
  • project — Link to Project Detail; the value is the project detail name.
  • assigned_to — Data.
  • deadline — Date; the JS colors it red if before today, orange if today, default otherwise.
  • status — Data.
Dynamic day columns

For each row the day column whose %d/%m matches the todo's deadline is filled with the todo's estimated value (as a string); other day cells are empty. The JS renders zero estimates in light grey.

How it works

Parameterized SQL joins tabProject Todo → tabProject Detail → tabProject. The WHERE clause requires todo.deadline IS NOT NULL, project.name = %(project)s, todo.status = %(status)s, and detail.is_pending = 0; when assigned_to is set it appends AND todo.assigned_to = %(assigned_to)s. Results are ordered by deadline, assignee, then estimated.

Roles: All System Manager

Missing project does not stop execution

If the project filter is missing, the report displays a prompt via msgprint but does not return early.

Todo Report

A comprehensive flat listing of todos belonging to currently Ongoing project details. It offers rich filtering (project, grouping, assignee, project-detail status, deadline range, todo status) and surfaces estimates, SOW notes, and todo notes for planning and review.

Reference doctype

Unlike the other four reports, this report's ref_doctype is the standard Frappe ToDo doctype, not Project Detail.

Filters

FilterTypeNotes
project — ProjectLink · ProjectFilters on pd.project.
assigned_to — Assigned ToLink · UserDefaults to frappe.session.user.
grouping — GroupingLink · GlossaryFilters on pd.grouping.
status — Project Detail StatusSelectOptions blank, Ongoing, Completed, On Hold; default Ongoing.
date_range — Todo Deadline BetweenDate RangeFilters pt.deadline between the range bounds inclusive.
todo_status — Todo StatusSelectOptions blank, ⚪️ Planned, 🟠 Done, 🔷 Checked By PL, ✅ Completed; default ⚪️ Planned.

Columns

Twelve fixed columns; no dynamic per-day columns.

Selected but not shown

The query selects more fields than it displays — project_detail_deadline, latest_todo, and expected_outcome are fetched but only the twelve fixed columns above are rendered.

How it works

The SQL joins tabProject Todo pt → tabProject Detail pd (pt.parent = pd.name) → tabProject p (pd.project = p.name) → tabGlossary g (pd.grouping = g.name). The base WHERE always enforces pd.status = 'Ongoing' AND pd.todo_count > 0, plus any dynamic filters. Results are ordered by deadline, detail name, then estimated, and each matched row is appended as a whole dict.

Roles: All System Manager Project Team Project Owner Project Leader

Base status constraint can conflict with the filter

The base WHERE always requires pd.status = 'Ongoing', so selecting Completed or On Hold in the Project Detail Status filter conflicts with that constraint and returns no rows.

SQL built by string interpolation

All six dynamic filters — project, grouping, assigned_to, status, date_range, and todo_status — are inserted into the query via direct f-string interpolation rather than bound parameters.

Project Progress dashboard

The app ships one default Dashboard, Project, displaying a single Dashboard Chart, Project Progress.

Project Progress chart

A standard, public Dashboard Chart in the Vernon Project module. It is a timeseries Line chart of chart type Sum that sums the estimated value of Project Todo records, bucketed by the deadline date field on a Daily interval over the Last Year timespan.

In effect, the chart plots the total estimated effort of ⚪️ Planned todos against their deadlines.

Project dashboard

The Project Dashboard is the app's default (is_default = 1) and standard dashboard. It carries no number cards and renders the single Project Progress chart at full width.
