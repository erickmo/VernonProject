API Reference

The four server-side endpoints Vernon Project exposes: one read on Project, two reads/writes on Project Todo, and the workflow-driving status updater. All are Frappe whitelisted methods callable from client JavaScript or over HTTP.

How these endpoints work

Every endpoint on this page is a Frappe whitelisted method. That means it can be invoked from server code, from client JavaScript via frappe.call, or directly over HTTP at /api/method/<dotted.path>. Because they are whitelisted, any logged-in user can reach them—authorization is decided inside each function, not by Frappe's route layer.

The two read endpoints (get_project_team_members and get_notes) perform no role checks at all. The two write endpoints (update_status and save_notes) save their document with ignore_permissions=True, which bypasses Frappe's document-level permission system entirely. To compensate, each write endpoint runs its own manual role check against the parent project's roles before it touches anything, and returns an error dictionary instead of saving when the caller is not allowed.

Writes bypass document permissions on purpose

Both write endpoints call todo.save(ignore_permissions=True). The only thing standing between a caller and a saved change is the manual allow-list inside the function. If you extend these endpoints, keep the manual check in place—Frappe will not enforce DocType permissions on these saves.

Quick reference

Method (dotted path)PurposeAuth

get_project_team_members

Returns the list of users assigned to a project's team. read

Defined in vernon_project/api/project.py. Loads the Project document and collects the user field from each row of its team_members child table.

Signature

python
@frappe.whitelist()
def get_project_team_members(project_name):
    ...

Parameters

NameTypeDescription
project_namestrThe name (docname / primary key) of the Project document to look up. required

Returns

A raw Python list of strings—the user value of each row in the project's team_members table, for example ["a@x.com", "b@x.com"]. If the project does not exist, the function catches frappe.DoesNotExistError and returns an empty list [].

No status wrapper

Unlike the Project Todo endpoints, this one returns the list directly—there is no {"status": ...} envelope and no {"message": ...} dict. A missing project and an empty team are indistinguishable: both yield [].

Permissions

None. No role or permission check is performed; ignore_permissions is not passed and there is no gate on project_leader, project_owner, or project_admin. Because it is whitelisted, any logged-in user can call it.

Example

js
frappe.call({
  method: "vernon_project.api.project.get_project_team_members",
  args: { project_name: "PRJ-25-06-00001" },
  callback: (r) => {
    // r.message is an array of user ids, e.g. ["a@x.com", "b@x.com"]
    console.log(r.message);
  },
});
bash
curl -s "$SITE/api/method/vernon_project.api.project.get_project_team_members" \
  -H "Authorization: token <api_key>:<api_secret>" \
  -G --data-urlencode "project_name=PRJ-25-06-00001"
# {"message": ["a@x.com", "b@x.com"]}

get_notes

Fetches the notes text of a single Project Todo. read

Defined in vernon_project/api/project_todo.py. Reads only the notes field straight from the database with frappe.db.get_value—it does not load the full document.

Signature

python
@frappe.whitelist()
def get_notes(todo_id):
    ...

Parameters

NameTypeDescription
todo_idstrThe name / ID of the Project Todo record whose notes should be fetched. required

Returns

Always a dict of the exact shape {'notes': <str>}. The value is the stored notes text, coerced to an empty string ('') when the stored value is falsy or None. There is no status key.

Because it uses frappe.db.get_value (not get_doc), an unknown todo_id does not raise DoesNotExistError; get_value returns None and the function returns {'notes': ''}.

Permissions

None. The function is explicitly documented as accessible by all logged-in users. There is no owner, leader, or admin gating.

Example

js
frappe.call({
  method: "vernon_project.api.project_todo.get_notes",
  args: { todo_id: cur_frm.doc.name },
  callback: (r) => {
    // r.message === { notes: "..." }
    field.set_value(r.message.notes);
  },
});

update_status

Advances a Project Todo one step through its four-stage status workflow. workflow

Defined in vernon_project/api/project_todo.py. Loads the todo, its parent Project Detail, and the grandparent Project, then branches on the todo's current status and the calling user's project role to perform exactly one transition.

Signature

python
@frappe.whitelist()
def update_status(todo_id):
    ...

Parameters

NameTypeDescription
todo_idstrThe ID / name of the Project Todo whose status should be advanced one step in the workflow. required

Returns

Always a dict with the keys status and message. The status value is only ever "error" or "info"—there is no "ok" here.

CaseReturned dict
Successful advance {"status": "info", "message": "Todo {to_do} is updated to {status}."}
Already ✅ Completed (terminal) {"status": "info", "message": "Todo {to_do} is already completed."}
Admin guard / permission denial / not found / generic error {"status": "error", "message": <reason>}
Success uses "info", not "ok"

A successful status advance returns status: "info"—not "ok". This is inconsistent with save_notes, whose success returns status: "ok". If your client code branches on the status string, treat "info" as success for update_status and "ok" as success for save_notes.

One control-flow subtlety: on the admin guard and on permission denials the function returns before saving. On the terminal ✅ Completed branch it returns the “already completed” info dict before saving too. On every genuine advance it calls todo.save(ignore_permissions=True) and then returns the success dict.

Permissions

The role check matches frappe.session.user against the parent project's project_leader, project_owner, project_admin, and the todo's assigned_to. The authority required narrows at each step:

TransitionWho may advance
⚪️ Planned → 🟠 DoneProject Leader OR Project Owner OR the todo's Assigned To
🟠 Done → 🔷 Checked By PLProject Leader OR Project Owner only assigned to cannot
🔷 Checked By PL → ✅ CompletedProject Owner ONLY leader cannot

Project Admin is hard-blocked. If the project's project_admin is set and the caller matches it, the function returns an error before evaluating any transition:

Project Admin tidak memiliki izin untuk mengupdate status todo. Silakan hubungi Project Owner atau Project Leader. (“Project Admin does not have permission to update todo status. Please contact the Project Owner or Project Leader.”)

The per-transition denial messages are mostly Indonesian/mixed. The exact strings are:

  • Planned denial: You do not have permission to approve this todo {todo.to_do} (Yg bisa hanya Project Owner {project_owner}, Project Leader {project_leader} atau Assigned To {todo.assigned_to}).
  • Done denial: You do not have permission to approve this todo {todo.to_do} (Yg bisa hanya Project Owner {project_owner}, Project Leader {project_leader}).
  • Checked By PL denial: You do not have permission to approve this todo {todo.to_do} (Yg bisa hanya Project Owner {project_owner}.
Known source quirks

The Checked-By-PL denial message has an unbalanced parenthesis in the source—it opens ( but never closes it. A missing todo returns {"status": "error", "message": "Todo {todo_id} does not exist."}, and any other unhandled exception returns {"status": "error", "message": str(e)}. The function uses error-return dicts throughout; it never calls frappe.throw.

Example

js
frappe.call({
  method: "vernon_project.api.project_todo.update_status",
  args: { todo_id: row.name },
  callback: (r) => {
    const res = r.message; // { status: "info" | "error", message: "..." }
    if (res.status === "error") {
      frappe.msgprint(res.message);
    } else {
      // "info" means success (advanced) or already-completed
      frappe.show_alert(res.message);
    }
  },
});

save_notes

Overwrites the notes field of a Project Todo. write

Defined in vernon_project/api/project_todo.py. Loads the todo, its parent Project Detail, and the grandparent Project, checks the caller against an allow-list, then assigns todo.notes = notes and saves.

Signature

python
@frappe.whitelist()
def save_notes(todo_id, notes):
    ...

Parameters

NameTypeDescription
todo_idstrThe ID / name of the Project Todo whose notes field is to be updated. required
notesstrThe new notes text to store on the todo (assigned directly to todo.notes). required

Returns

Always a dict with the keys status and message.

CaseReturned dict
Success{"status": "ok", "message": "Catatan berhasil disimpan."}
Permission denied{"status": "error", "message": <Indonesian reason>}
Not found{"status": "error", "message": "Todo {todo_id} tidak ditemukan."}
Generic error{"status": "error", "message": str(e)}
Success uses "ok" here—not "info"

Unlike update_status (whose success is "info"), a successful save_notes returns status: "ok". The success message is the Indonesian Catatan berhasil disimpan. (“Note saved successfully.”). Branch on the right success string per endpoint.

Permissions

The caller must be one of three roles: the todo's assigned_to, the project's project_owner, or the project's project_leader. Anyone else gets an error dict and nothing is saved. The save itself uses ignore_permissions=True, so this manual allow-list is the only thing governing access.

Project Admin is not on the allow-list

project_admin is not among the allowed roles, so a Project Admin cannot save notes unless they also happen to be the assignee, owner, or leader. The permission-denial message is: Anda tidak punya izin mengubah catatan ini. Yang boleh: Assigned To ({todo.assigned_to}), Project Owner ({project.project_owner}), atau Project Leader ({project.project_leader}). (“You do not have permission to change this note. Allowed: Assigned To, Project Owner, or Project Leader.”)

Example

js
frappe.call({
  method: "vernon_project.api.project_todo.save_notes",
  args: { todo_id: row.name, notes: editor.get_value() },
  callback: (r) => {
    const res = r.message; // { status: "ok" | "error", message: "..." }
    if (res.status === "ok") {
      frappe.show_alert(res.message); // "Catatan berhasil disimpan."
    } else {
      frappe.msgprint(res.message);
    }
  },
});

Status workflow recap

update_status is the only API entry point into the Project Todo state machine. The full lifecycle—side effects, recurrence, field locking, and deletion rules—is documented on the Todo Workflow page; this is the short version of who can push each gate.

⚪️
Planned
created
→
🟠
Done
assignee / leader / owner
→
🔷
Checked By PL
leader / owner
→
✅
Completed
owner only

Each update_status call advances the todo by exactly one step. The assignee, leader, or owner can push ⚪️ Planned to 🟠 Done; the leader or owner pushes 🟠 Done to 🔷 Checked By PL; and the owner alone pushes 🔷 Checked By PL to ✅ Completed. The escalation narrows toward the Project Owner at each gate, and the Project Admin can never advance status—they must contact the Owner or Leader. See Todo Workflow for the transition side effects and Permissions for the full role model.
