Roles & Permissions

Vernon Project layers custom code on top of Frappe's standard role permissions. Five roles, two server-side enforcement hooks, and a set of manual checks inside the whitelisted API together decide who can read, write, and advance work.

Roles

The app ships five effective roles. System Manager is the Frappe superuser role and bypasses every custom check. The four project roles (Project Owner, Project Leader, Project Admin, Project Team) are matched against the per-Project people fields (project_owner, project_leader, project_admin) and the team_members child table, not just the user's assigned Frappe roles.

Role vs. assignment

Having the Project Owner role is not the same as being a given project's owner. The API checks (and most has_permission logic) compare frappe.session.user against the project's project_owner / project_leader / project_admin field values and team membership. The Project Owner role mainly governs row-level list filtering and create access.

The API checks identity, not role

The update_status endpoint matches the calling user against the project's owner/leader/admin fields and the todo's assigned_to—it never inspects Frappe roles (except to whitelist the call). A System Manager who is not one of those people is also rejected by these per-transition gates. Only the DocType-level has_permission hooks and JSON permissions grant System Manager a blanket pass.

Two layers of enforcement

Beyond the static JSON role permissions, Vernon Project registers two of Frappe's server-side permission hooks in hooks.py. See Architecture for how these wire into the request lifecycle.

HookGranularityWhat it guardsRegistered for
permission_query_conditionsRow-levelInjects a SQL WHERE fragment that filters which rows appear in list views, reports, and queries.Project, Project Detail
has_permissionDocument-levelA per-document read/write check called when a single record is accessed.Project, Project Detail, Project Todo

Both hooks are wired in vernon_project/hooks.py. The permission functions are defined at module level in each DocType's controller (not as methods of the DocType class).

Row-level filtering (permission_query_conditions)

This hook runs only for list views, reports, and queries—it cannot apply to a child table loaded inside its parent.

Project

The Project query function returns an empty string (no restriction) when the user is empty/Guest, or when Project Owner is NOT among the user's roles. In other words, it restricts only users who hold the Project Owner role—every other role (Leader, Admin, Team, and users without that role) is left unrestricted by this function. For a Project Owner, the list is filtered to projects they own or are a team member of:

sql
`tabProject`.project_owner = '<user>'
OR EXISTS (
  SELECT 1 FROM `tabProject Team` pt
  WHERE pt.parent = `tabProject`.name
    AND pt.user = '<user>'
)

The user value is escaped via frappe.db.escape before being interpolated.

Project Detail

The Project Detail query function returns no restriction for empty/Guest users and for anyone with the System Manager role. Everyone else is limited to details whose parent Project lists them as owner, leader, admin, or team member:

sql
EXISTS (
  SELECT 1 FROM `tabProject` p
  WHERE p.name = `tabProject Detail`.project
    AND (
      p.project_owner  = '<user>'
      OR p.project_leader = '<user>'
      OR p.project_admin  = '<user>'
      OR EXISTS (
        SELECT 1 FROM `tabProject Team` pt
        WHERE pt.parent = p.name AND pt.user = '<user>'
      )
    )
)
Project Todo has no query conditions

Project Todo is a child table (istable: 1). Frappe does not apply permission_query_conditions to child doctypes, so there is no query function for it—its access is governed solely by the has_permission hook below.

Document-level checks (has_permission)

The has_permission functions decide read/write access for a single record.

Project

  • Returns True immediately if the user has the System Manager role.
  • If the user has the Project Owner role: create is always allowed; for any other permission type the user must be the document owner (doc.owner) or appear in the project's team_members.
  • Otherwise it returns False (including for users holding neither role).

Project Detail & Project Todo

  • Returns True for System Manager.
  • Otherwise the user must equal the project's project_owner, project_leader, or project_admin, or be one of its team_members.
  • For Project Todo the project is resolved through the chain: load the parent Project Detail (doc.parent), read its project, then load that Project. If the parent or its project is missing, access is denied.
Both Project Detail and Project Todo share the same access model

For both, anyone named on the parent Project (owner, leader, admin) or in its team can read and write the record. Note this document-level check makes no distinction between Project Admin and the others—the admin restriction lives in workflow logic, not here.

Inert permlevel-9 rows

Project Detail declares a second permission entry at permlevel 9 (read-only) for all four project roles. No field in the DocType is assigned permlevel 9, so these rows currently gate no fields and have no effect.

API-level permission checks

The whitelisted endpoints do not rely on Frappe's permission engine. Each performs its own manual role check and then saves with ignore_permissions=True, so only the in-function logic governs access. See API Reference for full signatures and return shapes.

EndpointManual checkProject Admin?
update_statusPer-transition: Planned→Done needs owner/leader/assigned_to; Done→Checked By PL needs owner/leader; Checked By PL→Completed needs owner only.Hard-blocked (early error return)
save_notesCaller must be the todo's assigned_to, the project's project_owner, or its project_leader.Not allowed (not in allow-list)
get_notesNone—documented as accessible to all logged-in users.Allowed (any user)
get_project_team_membersNone—any logged-in user may call it.Allowed (any user)
Admin can write the doc, but not advance it

Because the endpoints save with ignore_permissions=True, the role gates inside the function are the only barrier. save_notes deliberately omits Project Admin from its allow-list (only assignee, owner, leader)—so a Project Admin cannot save notes via the API even though the DocType-level permissions grant them write access. update_status blocks the admin before any other check.

Project Admin in depth

The Project Admin role (added in v0.4.0) is a write-capable role that is deliberately fenced off from the status workflow. The block is enforced in two independent places, so it holds whether status is changed through the API or directly on the form.

LayerWhereMechanism
APIupdate_status endpointA guard at the top: if project_admin is set and the caller equals it, it returns an error dict before any transition or save.
DocTypeProject Todo validate_project_admin_status_updateRuns during validate(). If the status changed and the current user equals the parent project's project_admin, it calls frappe.throw (title Permission Denied). System Manager is exempted.

Both layers raise the same Indonesian denial message:

text
Project Admin tidak memiliki izin untuk mengupdate status todo.
Silakan hubungi Project Owner atau Project Leader.

Glossed: "Project Admin does not have permission to update todo status. Please contact the Project Owner or Project Leader."

What Project Admin can do

Within projects where they are named, a Project Admin can read, write, and create Project Detail and Project Todo records (subject to the has_permission and row-level filters), and is auto-added to the project's team_members when set on the Project. They simply cannot advance a todo's status—those transitions must go through the Project Owner or Project Leader.
