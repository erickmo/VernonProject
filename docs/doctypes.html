DocType Reference

Field-by-field reference for all ten DocTypes in the Vernon Project app — three core work-management types and seven supporting types. Each entry lists its kind, naming, fields, controller logic, client scripts, and a pointer to the full permission matrix.

How to read these tables

Pure layout fields (Section Break, Column Break, Tab Break) are omitted. Required fields are marked required. Fields written by the server or framework rather than the user are noted as computed or auto-set. Some labels and validation messages are in Indonesian and are quoted verbatim. For the complete role/permission grid see Permissions.

Core DocTypes

The three core types carry the app's work-management workflow: a Project contains many Project Detail records, each of which owns a Project Todo child table.

Controller logic

  • validate — when both dates are set, throws Start Date cannot be after Deadline. if getdate(start_date) > getdate(deadline).
  • before_save — calls add_owner_and_leader_to_team() then remove_duplicate_team_members().
  • add_owner_and_leader_to_team() — appends owner, leader, and (if set) admin to team_members when not already present. It compares against a snapshot taken before the pass, so rows appended earlier in the same pass are not re-checked.
  • remove_duplicate_team_members() — keeps the first occurrence of each user and drops the rest.
  • Module-level get_permission_query_conditions(user) and has_permission(doc, ptype, user) — wired in hooks.py; restrict Project Owners to projects they own or are a team member of. See Permissions.

Client script

  • refresh sets a query on project_group returning {is_group: 0}, so only non-group Project Groups can be selected. No custom buttons, dialogs, or formatters.

Permissions: roles are System Manager, Project Owner, Project Leader, Project Admin, and Project Team; the Project Owner row is if_owner-restricted. Full matrix at Permissions.

Controller logic

  • before_validate — sets todo_count, computes total = price - discount, sets latest_todo to the max todo deadline, then calls recalculate_totals().
  • recalculate_totals() — rolls up todo_without_estimation, total_estimated, and total_remaining_estimated (Planned rows only), then sets status: Completed when total_remaining_estimated == 0 and todo_count > 0; else Pending when is_pending == 1; else Ongoing.
  • validate — guards required fields (Grouping is required., Project is required.), enforces that the grouping and each glossary belong to the selected project (Grouping must be part of the selected Project., Glossary <name> must be part of the selected Project.), and that price ≥ discount (Total SOW RP cannot be less than Total Discount.).
  • validate (delete-protection) — on existing docs, reloads the previous version and re-appends any todo whose status is not ⚪️ Planned, then throws Cannot delete Project Todo '<to_do>' unless its status is 'Scheduled'.
  • on_trash — blocks deleting the detail if any todo is not ⚪️ Planned: Cannot delete Project Detail because Project Todo '<to_do>' is not in 'Scheduled' status.
  • validate_assigned_to_team_member() — a helper that would reject assignees outside the project team, but it is not invoked by any controller hook in this file.
  • Module-level get_permission_query_conditions / has_permission — restrict access to the linked project's owner, leader, admin, or team members. See Permissions.
“Scheduled” means Planned

The delete-protection messages reference status 'Scheduled', but the only deletable status value in the codebase is ⚪️ Planned. The wording is a known source quirk — read “Scheduled” as “Planned”.

Client script

  • refresh filters grouping and glossaries options by {project}, sorts the todo rows in place (deadline asc, then estimated asc, then assigned_to asc), and triggers the project handler to apply the assignee filter.
  • refresh binds grid-row-render on the todo grid: it sets each row's allow_delete = false when status is not ⚪️ Planned (disabling the per-row delete button).
  • project resets grouping/glossaries when the project changes, then calls vernon_project.api.project.get_project_team_members and filters the todo grid's assigned_to options to project team emails.
  • price / discount recompute total client-side via calculate_total.
  • Inline Project Todo child events — action advances a row's status through a confirmation dialog (Yakin ubah status <to_do> ke <action>? — “Confirm changing the status of <to_do> to <action>?”) by calling update_status; before_todo_remove throws Cannot delete Project Todo unless its status is 'Scheduled'. for non-Planned rows. See Workflow for the full lifecycle.
Stray debug alert

The grid-row-render handler calls alert(status) for every todo row it renders — a leftover debug statement that pops a browser alert on form load. The action switch also assigns to an undeclared global action (no let/var). Both are harmless quirks worth knowing when debugging.

Permissions: System Manager, Project Owner, Project Leader, Project Admin (no delete), and Project Team (read-only). Permlevel-9 rows exist for the project roles but currently gate no fields. Full matrix at Permissions.

Controller logic (summarized — see Workflow · Phase tracking for depth)

  • validate — runs in order: validate_done_todo_fields(), validate_project_admin_status_update(), calculate_total_estimated_hours(), track_phase_changes(), then sets next_occurrence when recurring.
  • validate_done_todo_fields() — once status is 🟠 Done or ✅ Completed, locks assigned_to, estimated, and deadline; editing them throws (title Cannot Edit Completed Todo) Cannot modify <labels> when Todo status is '<status>'. These fields are locked once the todo is marked as Done or Completed.
  • validate_project_admin_status_update() — if the current user is the linked project's project_admin (and not a System Manager), blocks status changes with Project Admin tidak memiliki izin untuk mengupdate status todo. Silakan hubungi Project Owner atau Project Leader. (“Project Admin has no permission to update todo status. Please contact the Project Owner or Project Leader.”)
  • track_phase_changes() — sets the phase timestamp and computes the actual-hours field for each forward transition; calls calculate_total_actual_hours() at the end.
  • on_change — when the status changed, saves the parent Project Detail (cascading recalculation); when status is ✅ Completed and recurring, calls create_next_occurrence().
  • create_next_occurrence() — appends a new ⚪️ Planned todo to the parent with the next computed deadline (stopping past recurring_until) and msgprints Next recurring todo created with deadline: <date>.
  • on_trash — blocks deletion if linked to a Project Detail (Cannot delete Project Todo as it is linked to a Project Detail.) or if status is not Planned (Cannot delete Project Todo unless its status is 'Scheduled'.).
  • Module-level has_permission — resolves access through the parent's Project (owner, leader, admin, or team member). Note permission_query_conditions does not apply to child doctypes. See Permissions.

Client script

  • None — no project_todo.js exists in this doctype's folder. The Next button and status-advance UI are implemented on the parent Project Detail form.
Hooks note

The JSON permissions array is empty; access is governed at runtime by the module-level has_permission wired in hooks.py. The on_change entry in doc_events is commented out, but validate/on_change/on_trash are standard controller methods Frappe still invokes automatically.

Permissions: governed entirely by the runtime has_permission hook (no JSON permission rows). See Permissions and the API reference for update_status.
