Architecture

Vernon Project is a single Frappe app and module. This page maps its layered design — DocTypes and controllers, a whitelisted API, reports, permission hooks, a daily scheduled job, a client navbar customization, and a default dashboard.

Overview

Vernon Project ships as one Frappe app (vernon_project, title Vernon Project) exposing a single module, also named Vernon Project. Everything below lives inside that one module.

The app is organized in layers, each grounded in a Frappe primitive:

LayerWhat it doesWhere
DocTypes & controllersThe data model and its server-side business logic (validation, rollups, phase tracking, recurrence). See DocTypes.vernon_project/doctype/*
Whitelisted APIServer methods callable from the client for team lookups, notes, and status transitions. See API.vernon_project/api/*
ReportsFive operational reports for progress, todos, deadlines, and daily output. See Reports.vernon_project/report/*

Dashboard & chartA default Project dashboard rendering the Project Progress chart.dashboard_chart/, *_dashboard/

The data model

The model is a three-level work-breakdown chain. A Project owns one or more Project Detail records, and each Project Detail owns one or more Project Todo tasks. Project Todo is a standalone DocType (istable: 0) that links its parent through a project_detail Link; each todo splits its time across days via an allocations child table of Project Todo Allocation rows.

For the full picture see the entity-relationship diagram. Around the spine sit several supporting links:

  • Team. A Project holds a team_members child table of Project Team rows, each linking a User. The Project Owner, Project Leader, and (when set) Project Admin are auto-added to this table and de-duplicated by the controller.
  • People. A Project links project_owner, project_leader, and the optional project_admin to User. A Project Todo additionally links assigned_to plus the sign-off users developed_by, tested_by, and completed_by.
  • Classification. A Project links a brand to Brand (formerly Customer).
  • Scoring. A Project Todo links a group to Group and picks a level from that group's Group Level child rows, yielding a point value. Awards are written to Point Ledger entries — one row per role (assignee / leader).
  • Glossary. A Project Detail optionally links a grouping to Glossary and a glossaries Table MultiSelect of Project Glossary rows (each row links a Glossary). A Glossary is itself scoped to a Project, and the controller requires every selected glossary's project to equal the detail's own project. The grouping, when set, seeds a default scoring Group for new tasks.
  • Standalone stubs. Scope of Work and Project Proposal are submittable DocTypes that exist as skeletons, outside the Project chain (Scope of Work optionally links a Project Proposal).

App configuration & hooks

Integration points are registered in vernon_project/hooks.py. Four hook categories are active:

HookMappingPurpose
permission_query_conditionsProject → project.get_permission_query_conditions
Project Detail → project_detail.get_permission_query_conditions		Server-side list/report query filtering per DocType — restricts which records appear in list views and queries. See Permissions.
has_permissionProject, Project Detail, Project Todo → each DocType's has_permissionPer-document read/write access decisions on individual records. See Permissions.
app_include_js/assets/vernon_project/js/desk_navbar.jsInjects custom JS into desk.html on every desk page — here, the global navbar customization.
scheduler_eventsdaily → vernon_project.tasks.create_recurring_todosRegisters background jobs run by the Frappe scheduler.
No active document events

doc_events is defined in hooks.py but is an empty dict — its only entry (a Project Todo on_change → update_project_details handler) is commented out. No document events fire via this hook. Note that controller methods such as validate, on_change, and on_trash are still invoked by Frappe directly on each DocType class regardless of doc_events.

Scheduled jobs

One scheduled job is registered: vernon_project.tasks.create_recurring_todos, run daily under scheduler_events['daily']. It generates the next occurrence of completed recurring todos. See Recurring todos for the user-facing behavior.

Each run:

  • Selects due rows. A raw SQL query against tabProject Todo finds rows where is_recurring = 1, recurring_frequency is set, status = '✅ Completed', next_occurrence is not null and on or before today, and recurring_until is null or on or after today.
  • Skips duplicates (idempotency). For each due row it checks frappe.db.exists('Project Todo', {parent, to_do, deadline=next_occurrence, assigned_to}); if an identical next todo already exists it is skipped, so re-running the job does not create duplicates.
  • Appends the next todo. Otherwise it loads the parent Project Detail and source todo, computes the next date via calculate_next_occurrence(...), and (unless that date exceeds recurring_until) appends a new todo row copying the task details with status = '⚪️ Planned', then saves the parent with ignore_permissions=True.
  • Logs errors per row. Any exception on a single row is caught and logged via frappe.log_error(..., 'Recurring Todo Error'), and the loop continues with the next row.

After the loop, if any todos were created the job commits and logs an info message Created N recurring todos, then returns the count.

In effect the chart plots the total estimated effort of planned todos against their deadlines over the last year. The default Project dashboard (is_default = 1, standard) carries no number cards and renders this single chart at full width.

Client-side customization

The app_include_js hook loads public/js/desk_navbar.js globally — it runs on every desk page. On frappe.ready() the script selects the right-hand navbar container (.navbar .navbar-right) and prepends an <li> containing a Custom Menu link, adding that item to the desk top navbar.

Known issues in desk_navbar.js

The script contains two left-in placeholders. It fires a stray debug alert(1) on every desk page load, and the Custom Menu link points at a placeholder href /app/your-page, which is not a real route. Both should be removed or replaced before relying on this customization.
