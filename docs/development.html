Alasan, bukan referensi. Setiap catatan di bawah menjawab pertanyaan yang tidak bisa dijawab dengan membaca kodenya: kenapa dibangun begitu, dan apa yang rusak kalau diubah. Daftar DocType, endpoint, dan wiring ada di halaman Sistem — dirender dari kode, bukan diketik. Rationale, not reference. Every note below answers the question the code cannot answer for itself: why it is built this way, and what breaks if you change it. Kenapa beginiWhy it is built this way

Kenapa /w sengaja tidak punya service workerWhy /w deliberately has no service worker

Hanya /m yang mendaftarkan service worker (vernon_sw.js, scope /m): /m adalah PWA yang bisa dipasang — punya manifest, ikon, shell offline, dan push. /w tidak mendaftarkan apa pun, dan tidak ada file service worker kedua di repo ini. /m is the only app that registers a service worker (vernon_sw.js, scope /m): it is the installable PWA — manifest, icons, offline shell, push. /w registers nothing, and no second service worker file exists in this repo.

Itu bukan pekerjaan yang belum sempat dikerjakan. /w adalah aplikasi desktop di dalam tab yang memang selalu online; satu-satunya yang akan ditambahkan service worker di situ adalah lapisan cache di depan shell — persis benda yang catatan berikutnya ada untuk dimatikan. That is not an unfinished task. /w is a desktop app in a tab that is online by definition; the only thing a service worker would add there is a cache layer in front of the shell — precisely the thing the next note exists to switch off.

Kenapa no_store_spa_shell adaWhy no_store_spa_shell exists

Shell /m dan /w menunjuk file JS/CSS yang namanya mengandung hash isi. Setiap build menerbitkan hash baru dan menghapus yang lama. Frappe menyajikan HTML shell tanpa header Cache-Control, jadi browser boleh menyimpannya secara heuristik — dan shell lama yang tersimpan itu menunjuk ke JS yang sudah tidak ada di disk. Hasilnya: sesudah deploy aplikasi jadi layar putih, tanpa pesan error, dan hanya pulih setelah refresh manual. The /m and /w shells reference JS/CSS by content-hashed filename. Every build emits fresh hashes and deletes the old files. Frappe serves the shell HTML with no Cache-Control header, so a browser is free to cache it heuristically — and a cached OLD shell points at JS that no longer exists on disk. The result: after a deploy the app white-screens, with no error, and only recovers on a manual refresh.

Jebakannya: no_cache=1 di controller www hanya mematikan cache halaman redis di sisi server — ia tidak mengirim header cache HTTP sama sekali. Karena itu hook after_request menandai hanya HTML shell sebagai no-store; aset ber-hash di /assets tetap immutable dan boleh di-cache lama. Yang berubah kecil, yang besar tetap cepat. The trap: no_cache=1 in the www controllers only disables Frappe's server-side redis page cache — it emits no HTTP caching header at all. So an after_request hook marks only the shell HTML no-store; the hashed assets under /assets stay immutable and long-cacheable. The tiny thing revalidates, the big things stay fast.

Kenapa keterlambatan dihitung saat Done, bukan saat disetujuiWhy lateness clocks at Done, not at approval

Perhitungan poin membandingkan deadline dengan saat assignee menyerahkan pekerjaan — waktu todo dipindahkan ke Done — bukan saat leader menyetujuinya. Alasannya satu kalimat: review yang lambat tidak boleh menghukum orang yang sudah selesai tepat waktu. Kalau jam berjalan sampai persetujuan, denda keterlambatan assignee ditentukan oleh kalender orang lain. The point calculation compares the deadline against the moment the assignee handed the work off — when the todo moved to Done — not the moment a leader approved it. The reason is one sentence: a slow review must never penalize someone who finished on time. If the clock ran to approval, the assignee's late penalty would be set by someone else's calendar.

Persetujuan tetap menentukan kapan poin muncul — poin baru dicetak di Completed — tapi tidak pernah menentukan berapa. Todo lama yang tidak pernah mencatat timestamp Done punya rantai fallback, jadi tidak ada todo yang kehilangan perhitungan. Approval still decides when the points appear — they only mint at Completed — but never how many. Older todos that never recorded a Done timestamp fall back through a chain, so no todo loses its calculation.

Kenapa status Inbox sengaja status biasa, bukan modeWhy Project.status "Inbox" is deliberately a normal status

Inbox adalah salah satu opsi biasa di Project.status, sederajat dengan Ongoing dan Closed. Ia sengaja bukan flag terpisah dan bukan mode aplikasi: karena cuma label biasa, proyek Inbox tetap bisa dijadwalkan, dicari, difilter, dan diberi todo tanpa satu baris kode khusus. Inbox is an ordinary option on Project.status, a peer of Ongoing and Closed. It is deliberately not a separate flag and not an app mode: because it is just a label, an Inbox project stays schedulable, searchable, filterable and todo-able without a single line of special-case code.

Kuncinya: Inbox bukan Closed, tapi juga bukan Ongoing. Penentuan leader untuk persetujuan cuti mencocokkan Ongoing secara persis, jadi proyek Inbox terkecualikan dengan sendirinya — tanpa daftar pengecualian yang harus diingat orang saat menambah status baru. The point: Inbox is not Closed, but it is not Ongoing either. Leader derivation for leave approval exact-matches Ongoing, so Inbox projects exclude themselves — with no exclusion list for anyone to remember when a new status lands.

Kenapa skor, level, dan badge semuanya diturunkan dari Point LedgerWhy score, level and badge all derive from Point Ledger

Tidak ada penghitung yang disimpan di mana pun. Skor seumur hidup adalah penjumlahan baris Point Ledger milik user. "Jumlah todo selesai" adalah hitungan baris Point Ledger bersumber Todo dengan peran Assignee — bukan hitungan dokumen Project Todo. Poin badge adalah penjumlahan yang sama dengan sumber hadiah/hibah dikeluarkan. Semuanya kueri, bukan field. Nothing is stored as a counter. Lifetime score is the sum of a user's Point Ledger rows. "Todos completed" is a count of Point Ledger rows with source Todo and role Assignee — not a count of Project Todo documents. Badge points are the same sum with the reward/grant sources excluded. All of it is a query, not a field.

Konsekuensinya yang penting: tidak ada tally kedua yang bisa berselisih dengan yang pertama. Batalkan penyelesaian sebuah todo dan barisnya dicabut — skor turun sendiri, tanpa ada yang perlu "menghitung ulang". Dan reset bersih berarti menghapus baris ledger; tidak ada tempat lain yang diam-diam memberi poin kembali. The consequence that matters: there is no second tally that can disagree with the first. Un-complete a todo and its rows are pulled — the score falls out on its own, with nothing to "recalculate". And a clean reset means deleting ledger rows; nothing else quietly re-grants them.

Kenapa @ dibagi dua aplikasi sementara @web tidakWhy @ is shared across both apps while @web is not

Ada dua frontend: yang mobile melayani /m, yang web melayani /w. Di build mobile, @ menunjuk ke src miliknya sendiri. Di build web, @ menunjuk ke src milik mobile, dan @web menunjuk ke src web. Jadi alias itu bukan gaya penulisan — ia adalah batas kepemilikan. There are two frontends: the mobile one serves /m, the web one serves /w. In the mobile build, @ points at its own src. In the web build, @ points at the mobile app's src, and @web points at the web src. The alias is not a style choice — it is an ownership boundary.

Praktisnya: menyunting tipe atau hook di bawah @ mengubah kedua aplikasi sekaligus, dan bisa merusak build yang tidak sedang kamu buka. @web aman — hanya web. Ada jebakan kedua yang sudah terpasang penangkalnya: file bersama menarik dependensi dari node_modules milik mobile, sehingga tanpa dedupe ada dua salinan pustaka singleton (provider di salinan satu, hook membaca salinan dua). Jangan lepas dedupe itu. In practice: editing a type or hook under @ changes both apps at once, and can break the build you do not have open. @web is safe — web only. A second trap already has its guard in place: shared files resolve bare dependencies from the mobile app's node_modules, so without dedupe you get two copies of singleton libraries (the provider in one copy, hooks reading the other). Do not remove that dedupe.

Kenapa todo berulang punya dua jalurWhy recurring todos have two paths

Kejadian berikutnya bisa lahir dari dua tempat. Pertama: saat sebuah todo berulang mencapai Completed, controller-nya langsung mengantrikan penerus — supaya orang tidak perlu menunggu sampai besok untuk melihat pekerjaan berikutnya. Kedua: sebuah job harian menggulirkan setiap seri aktif ketika tanggalnya tiba — supaya seri yang tidak pernah diselesaikan tetap berjalan. The next occurrence can be born in two places. First: when a recurring todo reaches Completed, its controller queues the successor immediately — so nobody waits until tomorrow to see their next piece of work. Second: a daily job rolls every active series forward once its date arrives — so a series that is never completed still runs.

Keduanya memanggil generate_next yang sama, hanya beda satu flag: jalur on-complete memaksa, jalur scheduler hanya jalan kalau tanggalnya sudah tiba, sehingga kejadian masa depan tidak dibuat lebih awal. Keduanya mengunci akar seri lalu memeriksa duplikat, jadi dua jalur itu tidak bisa saling menerbitkan kembar. Ini yang perlu diingat: kejadian baru dibangun di satu tempat, dan field wajib yang baru harus ditangani di situ — kalau tidak, scheduler akan gagal diam-diam dan seri berhenti tanpa ada yang sadar. Both call the same generate_next, differing by one flag: the on-complete path forces, the scheduler path only fires once the date has arrived, so a still-open future occurrence is never pre-generated. Both lock the series root and then check for duplicates, so the two paths cannot race each other into twins. The thing to remember: the new occurrence is built in exactly one place, and any new required field has to be handled there — or the scheduler fails silently and the series just stops, with nobody noticing.

Kenapa path DocType bersarang tiga kaliWhy DocType paths are triple-nested

Sebuah DocType hidup di vernon_project/vernon_project/doctype/project_todo/: aplikasi, lalu modul, lalu doctype, lalu namanya. Nama yang berulang itu bukan salah ketik — yang luar adalah paket aplikasi, yang dalam adalah modul Frappe. Semua DocType di sini ada di satu modul yang sama. A DocType lives at vernon_project/vernon_project/doctype/project_todo/: app, then module, then doctype, then its own name. The repeated name is not a typo — the outer one is the app package, the inner one is the Frappe module. Every DocType here sits in that same single module.

Efek sampingnya nyata bagi siapa pun yang menghitung: satu modul untuk semuanya berarti pengelompokan klaster di halaman Beranda tidak ada di data mana pun — ia dipetakan tangan di dalam generator, di sebelah assert yang menolak DocType yang belum dipetakan. Dan menghitung direktori selalu berbohong: ada __init__.py dan __pycache__ yang ikut terjaring. Yang benar adalah membaca file JSON-nya dan menyaring apa isinya — persis yang dilakukan generator. The side effects are real for anyone counting: one module for everything means the cluster grouping on the Home page exists in no data at all — it is hand-mapped inside the generator, beside an assert that rejects any unmapped DocType. And counting directories always lies: __init__.py and __pycache__ get caught in the net. The honest way is to read the JSON files and filter on what is inside them — which is exactly what the generator does.

Kenapa situs ini tidak punya spanduk "terakhir diperbarui"Why this site has no "last updated" banner

Situs dokumentasi sebelumnya mati karena manusia mengetik fakta yang sudah diketahui kode, dan angka telanjang tidak bisa dibantah: pembaca tidak tahu apa yang dihitung, jadi tidak ada yang bisa menangkapnya saat salah. Karena itu satu aturan: kalau kode tahu, manusia tidak boleh mengetiknya — dan setiap angka tampil bersama glob yang menghasilkannya. Halaman Sistem seluruhnya dirender dari generator; halaman ini hanya berisi alasan, karena alasan adalah satu-satunya hal yang tidak bisa dibaca ulang dari kode. The previous docs site died because humans typed facts the code already knew, and a bare integer is unfalsifiable: no reader can tell what it counts, so no reader can catch it when it goes wrong. Hence one rule: if the code knows it, no human may type it — and every count ships with the glob that produced it. The System page is rendered entirely by the generator; this page carries only rationale, because rationale is the one thing you cannot read back out of the code.

Spanduk kesegaran sengaja tidak ada. Spanduk yang menyala di saat yang salah akan jadi hiasan dinding dalam dua bulan, dan tidak ada yang membacanya lagi justru ketika ia benar. Gantinya: jalankan ulang generator dan lihat apakah git melaporkan perubahan — kalau ada, situs ini tertinggal. Batas jujurnya: kalau tidak ada yang menjalankan generator, tier referensi basi secara diam-diam. Itu pertukaran yang disengaja, karena dengan nol fakta ketikan tangan, dokumen basi jadi tidak lengkap, bukan salah. There is deliberately no freshness banner. A banner that fires at the wrong moment is wallpaper by month two, and nobody reads it precisely when it is right. Instead: re-run the generator and see whether git reports a change — if it does, this site is behind. The honest ceiling: if nobody runs the generator, the reference tier goes stale silently. That is a deliberate trade, because with zero hand-typed facts stale docs are incomplete, not wrong.

Siklus hidup Project TodoThe Project Todo lifecycle

Ini satu-satunya alur kerja yang layak diberi prosa panjang, karena inilah aplikasinya. Project Todo adalah DocType terbesar di repo ini — paling banyak field, controller paling gemuk, lebih berat dari beberapa klaster lain digabung. Ukuran persisnya ada di halaman Sistem dan di mosaik klaster halaman Beranda, dihitung generator; di sini yang penting bukan angkanya, melainkan bentuk alurnya. This is the one workflow that earns real prose, because it is the app. Project Todo is the largest DocType in this repo — the most fields, the fattest controller, heavier than several other clusters combined. The exact sizes are on the System page and in the cluster mosaic on Home, counted by the generator; what matters here is not the number but the shape of the flow.

Dua tahap itu terpisah karena mereka menjawab dua pertanyaan berbeda. Done menjawab "kapan pekerjaannya selesai" dan dijawab oleh assignee. Completed menjawab "apakah pekerjaannya diterima" dan dijawab oleh orang lain. Menggabungkan keduanya akan memaksa satu timestamp menjawab dua pertanyaan — dan begitu itu terjadi, kecepatan review orang lain mulai mengubah nilai poin assignee. Those two stages are separate because they answer two different questions. Done answers "when was the work finished" and is answered by the assignee. Completed answers "was the work accepted" and is answered by someone else. Merging them would force one timestamp to answer both — and the moment it does, someone else's review speed starts changing the assignee's points.

Completed adalah satu-satunya tahap yang mencetak poin, dan itu yang membuat sisanya aman. Menandai Done tidak memberi apa-apa; menolak sebuah todo mengembalikannya ke Planned dan tidak ada yang perlu ditarik kembali, karena belum ada yang diberikan. Sebaliknya, membatalkan status Completed akan mencabut baris ledger yang tadi dicetak — jadi angka gamifikasi tidak pernah menyimpan hutang dari todo yang sudah tidak selesai lagi. Completed is the only stage that mints points, and that is what makes the rest safe. Marking Done grants nothing; rejecting a todo sends it back to Planned with nothing to claw back, because nothing was ever granted. Going the other way, un-completing pulls the ledger rows it minted — so the gamification numbers never carry a debt from a todo that is no longer finished.

Cancelled ada, tapi ia bukan tahap kelima: itu jalan keluar, bukan kemajuan. Sebuah todo bisa dibatalkan tanpa pernah lewat Done, dan tidak ada yang dicetak untuknya. Sementara itu Completed adalah pemicu yang juga mengantrikan kejadian berulang berikutnya — jadi satu perpindahan status itu mencetak poin, menutup pekerjaan, dan melahirkan yang berikutnya sekaligus. Cancelled exists, but it is not a fifth stage: it is an exit, not progress. A todo can be cancelled without ever passing through Done, and nothing mints for it. Meanwhile Completed is also the trigger that queues the next recurring occurrence — so that one status move mints the points, closes the work, and gives birth to the next one, all at once.

Catatan desainDesign log

Setiap spesifikasi dan rencana yang pernah ditulis, terbaru di atas, ditautkan bukan diringkas: baris bertanggal dalam bentuk lampau tidak bisa basi, sedangkan ringkasan adalah salinan yang perlahan berselisih dengan aslinya. Rencana diberi label skrip eksekusi — checkbox bukan status: itu instruksi kerja untuk agen, bukan bilah kemajuan, dan membacanya sebagai status akan menampilkan fitur yang sudah lama hidup seolah belum selesai. Every spec and plan ever written, newest first, linked rather than summarized: a dated past-tense row cannot rot, while a summary is a copy that slowly disagrees with its original. Plans carry the label execution script — checkboxes are not status: they are working instructions for an agent, not a progress bar, and reading them as status would show features that have been live for weeks as unfinished.
