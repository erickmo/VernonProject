Todo Workflow

The Project Todo lifecycle: a four-stage, role-gated status machine, the side effects each transition writes, per-phase time tracking, recurrence, field locking, and deletion rules.

The four-stage workflow

Every Project Todo moves through four ordered statuses. Each status carries an emoji prefix that must match exactly in code comparisons.

⚪️
Planned
created
→
🟠
Done
assignee / leader / owner
→
🔷
Checked By PL
leader / owner
→
✅
Completed

The states, in order, are ⚪️ Planned (the default), 🟠 Done, 🔷 Checked By PL, and ✅ Completed. The workflow is strictly linear: a single call to update_status() advances the todo by exactly one step, reading the todo's current status and the calling user's project role to decide whether the advance is allowed. There is no skipping ahead and no built-in way to move backward.

The authority required to advance narrows at each gate, escalating toward the Project Owner: the assignee can mark their own work 🟠 Done, a leader signs off the review into 🔷 Checked By PL, and only the owner performs the final acceptance into ✅ Completed.

Transition rules

The table below lists every transition the update_status() API performs, who may trigger it, and the audit fields it writes. The role check matches frappe.session.user against the parent project's project_leader, project_owner, project_admin, and the todo's assigned_to.

FromToWho may advanceSide effects
⚪️ Planned 🟠 Done Assignee OR Project Leader OR Project Owner Sets developed_at = now() and developed_by = current user
🟠 Done 🔷 Checked By PL Project Leader OR Project Owner assignee not allowed Sets tested_at = now() and tested_by = current user
🔷 Checked By PL ✅ Completed Project Owner ONLY leader cannot Sets completed_at = now() and completed_by = current user
✅ Completed Terminal. No transition occurs; the call returns an info dict, e.g. {"status": "info", "message": "Todo … is already completed."}

On a successful advance, update_status() returns {"status": "info", "message": "Todo … is updated to …"}. Note that success uses the status value "info", not "ok". A permission denial returns {"status": "error", "message": …} (mostly Indonesian) and the todo is not saved.

Project Admin is hard-blocked from all transitions

If the project's project_admin is set and the calling user matches it, update_status() returns an error before evaluating any transition: Project Admin tidak memiliki izin untuk mengupdate status todo. Silakan hubungi Project Owner atau Project Leader. (“Project Admin does not have permission to update todo status. Please contact the Project Owner or Project Leader.”) The same block is enforced independently at the DocType layer by validate_project_admin_status_update, so a Project Admin can never advance a todo—through the API, the grid button, or a direct save.

What happens on a status change

A status change does more than rewrite one field. When the Project Todo's status differs from its pre-save value, the controller's on_change hook loads the parent Project Detail and calls parent.save(). That cascading save re-runs the parent's before_validate rollups—todo_count, total_estimated, total_remaining_estimated, and the parent's own Pending / Ongoing / Completed status—so the detail always reflects the latest todo state.

If the new status is ✅ Completed and the todo is recurring, on_change additionally calls create_next_occurrence() to append the next row (see Recurring todos).

The Next button on the todo grid

In the UI, the Next button on a todo row triggers the advance. The parent form's child handler reads the current status, asks for confirmation—frappe.confirm('Yakin ubah status <to_do> ke <action>?') (“Are you sure you want to change the status of <to_do> to <action>?”)—and on Yes calls vernon_project.api.project_todo.update_status({todo_id: row.name}), then reloads the document. The form must be saved first; an unsaved form throws Please save the document before performing this action.

Phase time tracking

From v0.3.0, each todo captures both an estimate and an actual duration for all three workflow transitions, plus the timestamps that bracket each phase. Estimates are entered by the user; timestamps and actuals are computed automatically by the controller during validate.

PhaseEstimate field (you enter)Start → end timestamps (auto)Actual field (computed)
⚪️ → 🟠 Planned → Done estimated_planned_to_done planned_started_at → done_started_at actual_planned_to_done
🟠 → 🔷 Done → Checked By PL estimated_done_to_checked done_started_at → checked_started_at actual_done_to_checked
🔷 → ✅ Checked By PL → Completed estimated_checked_to_completed checked_started_at → phase_completed_at actual_checked_to_completed

Estimates

The three estimated_* fields are Float hours, optional, with 2-decimal precision. The controller method calculate_total_estimated_hours() sums them into total_estimated_hours, which is read-only and recomputed on every save.

Phase timestamps

The four Datetime fields are read-only and stamped automatically by track_phase_changes():

  • planned_started_at — set to now_datetime() when a new todo is created in ⚪️ Planned.
  • done_started_at — set on the transition to 🟠 Done (if not already set).
  • checked_started_at — set on the transition to 🔷 Checked By PL (if not already set).
  • phase_completed_at — set on the transition to ✅ Completed (if not already set).

Actuals

Each actual_* field is the elapsed time between its two bracketing timestamps, computed by calculate_hours_diff(start, end) as (end - start).total_seconds() / 3600, rounded to 2 decimals (and 0.0 if either timestamp is missing). The controller fills the relevant actual at the moment it stamps the closing timestamp, then calculate_total_actual_hours() sums all three into total_actual_hours.

Read-only totals

Both total_estimated_hours and total_actual_hours are computed/read-only—never edit them by hand. The legacy estimated (minutes) field is retained separately for backward compatibility and is independent of these per-phase hour fields.

Recurring todos

A todo can repeat on a schedule. The recurrence fields live in the collapsible Recurring Settings section:

FieldTypePurpose
is_recurringCheckEnables recurrence and on-completion creation of the next occurrence.
recurring_frequencySelectOne of Daily, Weekly, Monthly (or blank). Drives the next-occurrence math.
recurring_untilDateStop condition—recurrence ends once a computed next date would exceed this.
next_occurrenceDateRead-only; auto-computed in validate from deadline when empty.
original_todoDataRead-only, hidden; carries the source row name so the recurring chain stays linked.

Next-occurrence math

The helper calculate_next_occurrence(from_date) takes the base date (the todo's deadline, defaulting to today if absent) and returns:

FrequencyNext date
Dailyadd_days(from_date, +1)
Weeklyadd_days(from_date, +7)
Monthlyadd_months(from_date, +1)

If recurring_until is set and the computed next date is later than it, recurrence stops and no new row is created.

Two creation paths

(a) On completion. When a recurring todo reaches ✅ Completed, on_change calls create_next_occurrence(). It appends a new todo row to the parent Project Detail—copying to_do, assigned_to, estimated, and notes, carrying over the recurrence settings, setting deadline to the next date, original_todo to the chain root, and status back to ⚪️ Planned—then saves the parent and shows Next recurring todo created with deadline: ….

(b) Daily scheduled job. A daily job, create_recurring_todos, finds completed recurring todos whose next_occurrence date has arrived (status ✅ Completed and next_occurrence ≤ today) and creates the follow-up row if it was not already generated—for example, if the job did not run on the day the todo completed. It is idempotent, so running it repeatedly does not produce duplicates. See Scheduled jobs for how it is wired into scheduler_events.

Locked fields once Done or Completed

The controller method validate_done_todo_fields (v0.2.0) freezes the planning fields once a todo reaches 🟠 Done or ✅ Completed. It skips brand-new rows; for an existing row in a locked status, it compares each protected field against get_doc_before_save() and rejects any change.

These fields are locked once the todo is marked as Done or Completed.

Fields outside that set—such as notes and to_do—remain editable in every status. The rationale is audit and reporting integrity: once work is signed off, the recorded assignee, estimate, and deadline must stay fixed so historical and performance reports stay trustworthy.

Deletion rules

A Project Todo can only be deleted while it is still in ⚪️ Planned. Two layers enforce this:

  • The todo itself. on_trash blocks deletion of any todo whose status is not ⚪️ Planned (and also blocks deletion while the row is still linked into a Project Detail).
  • The parent Project Detail. Its on_trash refuses to delete the detail when any of its todos is past ⚪️ Planned. On save, the detail also compares its todos against the previous version: if a non-Planned todo was removed from the grid, it re-appends the row and throws, restoring the illegally-removed task.

In the UI this is reinforced by the grid: the per-row delete button is disabled for any todo whose status is not ⚪️ Planned, and before_todo_remove throws if you try anyway.

“Scheduled” in the message vs. the real status

The deletion error messages say the status must be 'Scheduled' — for example Cannot delete Project Todo unless its status is 'Scheduled'. No such status exists. The actual deletable status is ⚪️ Planned; Scheduled is leftover wording. Read it as “must be Planned.”
